DEVELOPER-LEVEL SPECIFICATIONS

DATA ACCESS CLASSES

The data access classes are to be used whenever data needs to be obtained from a system external to the DME UI. This includes access to the “Core 2” configuration engine.


This particular section does not reference any Core transactions specifically; however, most of the “data access” classes do directly reference Core transactions. For more information on what classes use Core transactions, see section 4.2.14.


These classes offer convenience routines that are in place so that implementations can be easier swapped down the road. In addition to providing the flexibility to change functionality, these classes make data-access maintenance significantly easier for the developer.

1.)     DAArrangement


         The DAArrangement class is the data access class to use to access information relating to arrangements. This class can retrieve a list of arrangements or

         an individual arrangement. In addition to these two pieces of functionality, the class can do much more. Core transactions are used by this class.


2.)     DAAudit


         The DAAudit class is the data access class to use for running audits. This class is not yet “mature”, as much of it is still in flux.


3.)     DAChange


         The DAChange class is the data access class to use for obtaining Change Management data. This information is used in the “Change Picker” in the

         application. The requests from this class call the BWS “Shared Libraries” that is associated with the Core 2 Engine.


4.)     DADetail


         The DADetail class is the data access class to use for querying or updating detail information.  Core transactions are used by this class.


5.)     DADrawing


         The DADrawing class is the data access class to use for querying or updating drawing information. Core transactions are used by this class.


6.)    DAEdition


         The DAEdition class is the data access class to use for querying or updating edition information. Core transactions are used by this class.


7.)     DAException


         The DAException class is the data access class to use when exceptions are deemed to occur while the user is requesting data. Typically, this exception

         serves as a wrapper of the Core transaction’s “TranResult”. Whenever data access methods are called, the developer must be prepared to catch and

         handle DAExceptions—otherwise unanticipated crashes may occur.


8.)     DAImportExport


         The DAImportExport class is the data access class to use for exporting or importing data.


9.)     DAMarking


         The DAMarking class is the data access class to use for loading data marking information for arrangements. This information is currently being pulled by

         directly using SQL and querying the database.


10.)     DAModel


           The DAModel class is the data access class to use for manipulating data at the model level. For example, there is a method available to wipe out all of the

           data for a particular model. Core transactions are used by this class.


11.)     DAPDM


          The DAPDM class is the data access class to use for querying PDM data from R2K databases. The R2K databases are Oracle based. This class

          references methods that are in the BWS_ServiceLibrary project. Right now, that code is just deployed as an assembly; later on, it is desirable to wrap this

          logic in a portable webservice.


12.)     DAQuery


           The DAQuery class is the data access class to use for querying the NWS database, either by direct SQL or by using the Core’s “AQL”. AQL (Advanced

           Query Language) is a set of wrapper classes that enforce metadata constraints and perform SQL-like functions.


13.)     DAReserve


           The DAReserve class is the data access class to use for querying Reservation data.This class directly calls the DalRI class in the BWS_Base assembly.
